# SBUS 兼容性与常见陷阱

## 本项目所说的“标准 SBUS”

除非另有说明，本项目的 SBUS 指以下组合：

- 单向控制数据；
- `100000 bit/s`、8 数据位、偶校验、2 停止位（`8E2`）；
- 线路侧为反向 UART 逻辑；
- 每帧 25 字节，帧头 `0x0F`；
- 16 路、每路 11 bit，Flags 位于第 24 字节；
- 标准帧尾 `0x00`。

“支持 SBUS”“兼容 SBUS 舵机”“可以接到标有 SBUS 的引脚”都不足以证明上述条件成立。兼容性可能只发生在设备功能层、连接器层或电气层，未必是协议字节级兼容。

## 兼容性矩阵

| 名称或模式 | 与标准 SBUS 的关系 | 当前上位机 | 常见陷阱 |
| --- | --- | --- | --- |
| 标准 SBUS / SBUS-16 | 本项目基准 | 完整支持 | 仍需确认反相逻辑、`100000 8E2` 和 25 字节帧 |
| 非反相 SBUS | 控制帧可相同，物理极性不同 | USB→RS-422 解码后可分析；本项目硬件输入默认要求反向 SBUS | “数据格式相同”不等于可以直接接线 |
| S.BUS2 | 在 S.BUS 基础上增加双向通信和遥测用途 | 识别 RC 控制帧及已知 S.BUS2 帧尾；不解析遥测时隙 | 控制帧能解码，不代表完整 S.BUS2 总线已经透明通过或通过验收 |
| W.BUS / W.BUS2 | 厂商资料可能称兼容 S.BUS 信号或舵机，但不应据此推定字节流完全相同 | 不作为标准 SBUS 接受 | 项目测试曾观察到循环变化的扩展帧尾和附加字节；舵机兼容是单向功能兼容，不是严格协议等价 |
| SBUS-24 | 从 SBUS-16 扩展出的 24 通道变体 | 不支持 | 只有链路上所有设备都支持该模式时才能使用；固定 16 通道解析器不能据名称自动兼容 |
| F.Port / F.BUS | 把控制与遥测合并到单线总线的另一类协议 | 不支持 | F.Port 使用 `115200 8N1`，与 `100000 8E2` SBUS 不同；F.BUS 也不能当作 SBUS 帧流 |
| CRSF / 反相 CRSF | 独立的双向串行协议 | 不支持 | 反相 CRSF 接到“SBUS”标记引脚后仍然是 CRSF；电平反相不会改变协议 |
| 厂商修改版或高速 SBUS | 可能沿用部分 SBUS 帧结构，但波特率、限位、通道或时序不同 | 默认不支持 | 名称包含 SBUS 也必须重新核对波特率、校验、帧长、帧尾、通道数和极性 |
| iBUS、SUMD、SRXL2、GHST 等 | 其他遥控串行协议 | 不支持 | 接收机可能同时支持这些协议和 SBUS，必须确认当前端口实际选择的输出模式 |

## W.BUS 踩坑记录

项目测试中遇到过标称可兼容 S.BUS 的 W.BUS 输出。它的控制数据具有明显的 SBUS 相似性，但捕获中出现了在 `0x01–0x0F` 间循环变化的扩展帧尾和附加字节，因此不能作为标准 25 字节 SBUS 流进行“零改动”验收。

这不等于 W.BUS 设备或 S.BUS 舵机不能配合使用。它说明的是：

1. 舵机能够响应通道数据，只证明设备功能层兼容；
2. 标准 SBUS 接收端可能忽略扩展部分，所以表面上工作正常；
3. 严格的透明桥接验证比较所有字节和时序，扩展字节不能被忽略；
4. 如果产品目标是标准 SBUS，应切换到明确标注为 S.BUS/SBUS-16 的输出模式并重新采集确认。

## S.BUS2 的支持边界

S.BUS2 扩展了传统 S.BUS，并用于双向通信和遥测设备。当前解析器接受 `0x00`、`0x04`、`0x14`、`0x24`、`0x34` 这些控制帧尾，并会把非 `0x00` 的已知帧尾标记为 S.BUS2。

但本项目当前不会解释控制帧之外的遥测时隙。因此：

- 可以验证其中的 16 路 RC 控制帧；
- 不能据此证明遥测传感器数据正确；
- 出现在控制帧之间的遥测数据可能被统计为非控制帧字节；
- 要证明 S.BUS2 全总线透明传输，必须另行保存并逐字节比较完整原始总线数据。

## 排查顺序

遇到“看起来像 SBUS 但无法稳定解析”时，依次检查：

1. 端口当前配置的实际输出协议，而不是端口丝印或接收机支持列表；
2. 波特率、数据位、校验和停止位；
3. 正常或反相逻辑，以及适配器是否已经做过一次反相；
4. 帧长、帧头、帧尾和通道数；
5. 帧间是否插入遥测、配置或厂商扩展字节；
6. `FRAME LOST`、`FAILSAFE` 与厂商扩展标志位的定义；
7. 用逻辑采集文件同时比较输入和输出，不以“舵机会动”替代逐字节证明。
