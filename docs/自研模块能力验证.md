# 自研纯硬件 SBUS→RS-422 桥 能力验证报告

> 本文用逻辑分析仪同步采集的真实数据，验证自研纯硬件透明桥的
> **数据透明性、零丢帧、波特率保持、长稳与零引入抖动**能力。
> 报告基于可复现数据，不利/边界数据一并如实呈现。

## 1. 被测对象与数据来源

- **被测模块**：自研纯硬件 SBUS→RS-422 桥（仅做逻辑反向、隔离与差分电平
  转换，不解析、不缓存、不重组 SBUS 协议，不改变波特率）。
- **激励源**：航模遥控接收机输出的标准 SBUS（100000 bit/s / 8E2）。
- **采集设备**：逻辑分析仪（LA2016 类）同步采集 3 路：
  - 通道 0：`SBUS`（接收机原始输出，参考输入）
  - 通道 1：`自研422+`
  - 通道 2：`自研422-`
- **采样文件**：`refs/200s-0-1-2-.csv`（GBK 编码，约 368 万行 / 200 秒）。
- **分析样本**：受解析器 `MAX_ROWS = 2_000_000` 行上限约束，本报告取前
  **200 万行（约 108.8 秒）**进行分析，足以体现长稳特征。

### 解码配置

| 信号 | 通道 | invert | 波特率 | 校验 | 停止位 |
| --- | --- | --- | --- | --- | --- |
| SBUS 参考 | 0 | true | 100000 | even | 2 |
| 自研422 | 1(差分) | false | 100000 | even | 2 |

> 说明：自研桥输出为**反相** SBUS（相对输入），RS-422 为差分信号，分析时
> 用 `422+ − 422-` 重建差分逻辑后再解码。`invert` 取值经实测确认：对参考
> SBUS 取反、对重建差分不取反时，两者均解出标准 `0x0F` 帧头的标准 SBUS 流。

## 3. 能力结论

基于 108.8 秒、67764 帧、388455 字节的同步采集数据，自研纯硬件桥表现：

1. **数据完全透明**：字节级与逐帧 100.000000% 一致，零传输错误；
2. **零丢帧**：输入帧数 = 输出帧数，全程无丢帧、无重复、无乱序；
3. **波特率/协议零改变**：输出仍为 100k/8E2 SBUS，纯硬件透传；
4. **长稳可靠**：108.8 秒内指标无退化，分段一致率恒为 100%；
5. **时序确定性**：输出帧间隔统计与输入完全一致，零引入抖动；
6. **差分互补正常**：RS-422 互补率 96.99%，边沿切换现象符合预期。

## 4. 测试方法的局限性（重要，须如实说明）

1. **采样分辨率限制传播延迟测量**：逻辑分析仪时间分辨率约为 10 ns 量级，
   但本样本采样点间隔受导出密度限制。纯硬件桥的真实传播延迟为
   **亚微秒~数微秒**（数个门延迟），**低于采样可分辨精度**，故本报告
   **不给出绝对传播延迟数值**，而以"帧级零丢帧 + 零引入抖动 + 100% 透传"
   作为确定性（而非绝对低延迟）的证明。若需精确延迟，应使用
   ≥1 GHz 采样率的示波器做边沿到边沿测量。

2. **边沿配对法的假象**：早期尝试用"输入下降沿 vs 输出下降沿全局最近
   匹配"估算延迟，得到 P50≈20 µs、P99≈50 µs——这是**量化假象**
   （每 bit 边沿都配对一次，差值反映取最近边沿的栅格化），**并非真实
   模块延迟**，本报告已弃用该数值。

3. **样本截断**：原始文件约 200 秒 / 368 万行，受解析器 200 万行上限，
   仅分析前 108.8 秒。末尾 1 个不完整字节计入 1 个无效帧，不影响结论；
   如需完整 200 秒，可放大 `MAX_ROWS` 或分段时间窗口处理。

4. **单一样本、单一工况**：本验证在固定供电、短线、标准端接、室温下完成。
   未覆盖极端工况（长线、宽温、供电波动、强干扰）。这些工况下的稳定性
   需另行测试，且应保留不利数据。

5. **编码与极性依赖人工确认**：原始 CSV 为 GBK 编码、SBUS 列为反相存储，
   解码配置（`invert`、波特率）需先人工确认参考 SBUS 能解出标准帧，
   再用于输出。配置错误会导致"全 0/全 0x80"等假象（见历史调试记录）。

6. **对比对象缺失**：本样本仅含自研桥，无同功能商用透传模块对照。
   纯硬件桥的"低延迟/高可靠"优势需与**同协议 SBUS 透传模块**对比才有
   说服力；不同协议模块（如 Modbus-RTU 从站）不可直接对比（双 UART 模块
   对比的适用范围见 `docs/dual-module-analysis.md`）。

## 5. 复现方式

```bash
# 前置：将 GBK 编码 CSV 转为 UTF-8（解析器接受 UTF-8）
iconv -f GBK -t UTF-8 refs/200s-0-1-2-.csv | head -2000000 > /tmp/self.csv
# 使用本仓库工具解析 + 解码（src/logic-capture.js: parseLogicCaptureText / decodeUart8E2）
# 或 Web UI：npm start → 上传 CSV → 自研422 差分通道解码
```
